# Release checklist

Use this checklist before creating a public binary release.

## Local verification

Run:

```bash
cargo fmt --all -- --check
cargo test --locked
cargo clippy --all-targets --locked -- -D warnings
cargo build --release --locked
```

Then manually smoke-test the app:

```bash
cargo run --release
```

Check at least:

- the app opens successfully;
- the window can be resized and scrolled;
- characteristics can be rolled and edited;
- age controls update derived stats;
- occupation choices unlock skills correctly;
- skill allocation warnings display correctly;
- JSON save copies to the clipboard;
- pasted JSON loads into a fresh app state;
- JSON can be saved to and loaded from a file path;
- import correction messages appear when invalid allocations, custom skill labels, Luck evidence, or legacy save fields are sanitized;
- loaded investigators preserve characteristics, occupation choices, custom skill labels, duplicate custom slots, active and inactive custom-slot allocations, skills, Luck, RNG seed/roll history, age modifiers, and backstory;
- summary text copies to the clipboard.

## Creating a release candidate

For the first packaging pass, prefer a release-candidate tag before the final version tag:

```bash
git tag v0.1.0-rc1
git push origin v0.1.0-rc1
```

The release workflow builds raw archives and installer packages, then creates or updates the GitHub Release for the tag.

## Artifact status and smoke testing

The Linux raw `.tar.gz` and Linux `.AppImage` are the maintainer-verified artifacts when tested locally on a non-Debian distribution.

Other package formats are generated by GitHub-hosted runners and should be treated as experimental until someone smoke-tests them on the matching platform:

- Linux `.deb`: generated by CI, not locally install-tested on non-Debian systems;
- Windows `.zip` and NSIS `.exe`: generated by CI, experimental until tested on Windows;
- macOS `.tar.gz`, `.app.tar.gz`, and `.dmg`: generated by CI, experimental until tested on macOS;
- Linux Pacman package output, if produced by `cargo-packager`: generated by CI, experimental until tested on Arch-compatible systems.
- CI pins `cargo-packager` to version `0.11.2`; update the workflow and this note together when intentionally changing packaging tool versions.

For release notes, say clearly which artifacts were manually tested and which are experimental. Do not describe unsigned Windows/macOS installers as polished end-user packages until signing/notarization and platform smoke tests are complete.

## Checksums and attestations

The release workflow generates `SHA256SUMS.txt` after all raw archives and installer packages are downloaded into the release job. The checksum file is uploaded to the GitHub Release beside the artifacts.

The release workflow also generates GitHub artifact attestations from `SHA256SUMS.txt` before uploading release assets. The checksum file is used as the subject list, so each artifact named in it gets provenance tied to the workflow run that gathered and checksummed the release assets.

To verify checksums after downloading artifacts:

```bash
sha256sum -c SHA256SUMS.txt
```

To verify GitHub artifact attestations, first verify the downloaded files against `SHA256SUMS.txt`, then use the GitHub CLI from a checkout or provide the repository explicitly for each downloaded artifact:

```bash
gh attestation verify <downloaded-artifact> --repo P2949/Coc-app
```

## Creating a final release

Once the release-candidate artifacts look acceptable, update `Cargo.toml` and `Packager.toml` to the final release version if needed, then tag the final version:

```bash
git tag v0.1.0
git push origin v0.1.0
```

## Versioning

Update `Cargo.toml` before tagging a new release:

```toml
version = "0.1.0"
```

Update `Packager.toml` to the same version:

```toml
version = "0.1.0"
```

Use tags that match the version, for example:

```text
v0.1.0
```

## Code signing status

The current release workflow produces unsigned artifacts.

That is acceptable for an early experimental release aimed at technical users, but it is not ideal for broad public distribution:

- Windows: an unsigned NSIS installer may trigger SmartScreen warnings. Future work should add Authenticode signing with a code-signing certificate.
- macOS: unsigned and unnotarized `.app.tar.gz` / `.dmg` artifacts may be blocked or heavily warned by Gatekeeper. Future work should add Developer ID signing and notarization.
- Linux: AppImage and package signing can be added later if you want stronger distribution trust outside GitHub Releases.

Do not claim Windows or macOS installers are fully polished until signing/notarization and platform smoke tests are done.

## Notes

The release workflow uses host-native builds on GitHub-hosted runners. It does not cross-compile.
